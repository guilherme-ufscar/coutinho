## Fase 0 — Fundação

- **Gerenciador de pacotes**: npm workspaces (monorepo simples), em vez de pnpm/turborepo — reduz dependências externas de bootstrap; pode migrar para Turborepo em fase posterior se o build ficar lento.
- **Dev domains**: Caddy local usa `*.localhost` (ex.: `app.localhost`, `api.localhost`) em vez do domínio de produção, para HTTPS automático local sem precisar de DNS público. Produção usa `couthealth.com.br` / `app.couthealth.com.br` / `api.couthealth.com.br` via `.env`.
- **API NestJS na Fase 0**: sobe como esqueleto mínimo (health endpoint) — módulos de domínio completos entram nas fases 3+ conforme o roadmap.
- Demais defaults: ver `escopo.md` §13.0 (fonte Space Grotesk/Inter, PaymentProvider mock+Asaas, YouTube não listado, MinIO, papéis client/professional, etc.) — todos adotados sem alteração.
- **`.env` local**: `DOMAIN`/`APP_DOMAIN`/`API_DOMAIN` default para `*.localhost` (não os domínios de produção) para que o Caddy emita certificado interno automaticamente em dev. `.env.example` documenta a troca para produção.
- **Next.js standalone em Docker**: obrigatório setar `ENV HOSTNAME=0.0.0.0` no Dockerfile — o Docker define `HOSTNAME` automaticamente como o ID do container, e o `server.js` gerado pelo `output: "standalone"` usa `process.env.HOSTNAME` para bind, fazendo o healthcheck falhar silenciosamente (`connection refused` em 127.0.0.1) sem esse override.
- **Backup do Postgres**: trocado cron (`dcron`) por um loop `while true; do backup; sleep 86400; done` — `crond -f` falhava com `setpgid: Operation not permitted` dentro do container (sandboxing). Mais simples e portável para o MVP.
- **Fase 1 (Stitch)**: o Stitch MCP (`@_davideast/stitch-mcp` proxy) segue `✔ Connected` no `claude mcp list`, mas suas ferramentas (`create_project`, `generate_screen_from_text` etc.) não aparecem no `ToolSearch` desta sessão — MCPs adicionados no meio de uma sessão do Claude Code só carregam ferramentas após reiniciar. Sem reinício disponível, a Fase 1 é **pulada como etapa isolada**: as telas são implementadas direto em código a partir do Prompt Base/Design DNA (§7) e dos prompts por tela (§9) de `design-stitch.md`, usados como especificação textual em vez de mockup gerado. Retomar a geração real no Stitch em uma sessão futura é possível a qualquer momento (chave e conexão já configuradas).
- **Validação Playwright da Fase 0**: parcial. Playwright MCP foi instalado no meio desta sessão do Claude Code; o cliente MCP só carrega ferramentas de servidores novos após reiniciar a sessão, então `browser_navigate`/`browser_take_screenshot` etc. não ficaram disponíveis neste turno. `claude-in-chrome` também indisponível (extensão do Chrome não conectada no ambiente). Validação da Fase 0 foi feita via `curl` (status HTTP 200 + payload correto nos 3 domínios). Repetir a validação visual completa (screenshots, console) na próxima fase com UI, quando um dos dois caminhos estiver disponível.

## Fase 2 — Landing (3 variantes)

- **`NEXT_PUBLIC_APP_URL`**: Next.js inlina variáveis `NEXT_PUBLIC_*` no momento do **build**, não do runtime. Passado como `build.args` no `docker-compose.yml` (derivado de `APP_DOMAIN`) em vez de `environment:`, senão o CTA "Criar conta" ficaria hardcoded para `app.localhost` mesmo em produção.
- **Botões de CTA como `<a>`** (não `onClick`): o componente `Button` do design system virou polimórfico (`href` → `<a>`, senão `<button>`) para que as 3 landings continuem Server Components (SEO/performance) — só as seções interativas (`PlansSection` com toggle de período, `FaqSection` com accordion) são Client Components isoladas.
- **`VariantSwitcher`**: navegação flutuante temporária (`/`, `/v2`, `/v3`) nas 3 landings, só para o cliente comparar as direções antes de escolher uma — remover depois da escolha (DoD da Fase 2 no `escopo.md`).
- **Validação Playwright/browser da Fase 2**: mesmo bloqueio da Fase 0 (ferramentas do Playwright MCP não carregadas nesta sessão; `claude-in-chrome` sem extensão conectada). Validação feita via `curl` (status HTTP das 3 rotas) + `next build` sem erros de tipo/lint. Screenshots desktop/mobile pendentes — repetir assim que um caminho de browser real estiver disponível.

## Fase 3 — Auth + Planos + Pagamento

- **Prisma 7 → 5**: `npm install prisma@latest` trouxe a v7, que quebra o padrão `datasource { url = env("DATABASE_URL") }` no schema (exige `prisma.config.ts` + driver adapters). Fixado em `^5.20.0` (estável, compatível com o padrão usado por NestJS/Prisma na maioria dos tutoriais e com o restante do setup) em vez de migrar para o novo modelo de config — reavaliar quando o ecossistema NestJS+Prisma 7 amadurecer.
- **`npm approve-scripts`**: o npm 11 bloqueia scripts de instalação de pacotes novos (argon2, prisma, esbuild) por padrão. Aprovados via `npm approve-scripts --all` (gravado em `allowScripts` no `package.json` raiz) — necessário pro argon2 compilar o binário nativo e o Prisma rodar seus scripts de setup.
- **Dockerfile da API sem `--ignore-scripts`**: diferente dos outros Dockerfiles do monorepo, o da API roda `npm install` completo (scripts habilitados) porque o argon2 precisa compilar via node-gyp (`python3 make g++` instalados no estágio `base`) e o Prisma precisa rodar `prisma generate`. As aprovações já estão persistidas no `package.json`, então não há prompt interativo no build.
- **`docker-entrypoint.sh` da API**: roda `prisma migrate deploy` + `prisma db seed` (idempotente, `|| true`) antes de subir o Nest — garante que um ambiente novo (`docker compose up` do zero) já sobe com schema aplicado e os 3 planos/cupom/admin seedados, sem passo manual.
- **Postgres exposto em `127.0.0.1:5432`** (loopback-only) no `docker-compose.yml`: necessário para rodar `prisma migrate dev` do host durante o desenvolvimento. Mesma instância/volume usada pelo container `api` internamente (`postgres:5432`) — não é um banco separado. Remover/restringir antes de produção (ver Fase 10).
- **Google OAuth sem credenciais reais**: `GoogleStrategy` usa placeholders (`"not-configured"`) em vez de deixar `clientID`/`clientSecret` vazios, porque o `passport-oauth2` lança exceção na construção do provider com string vazia — isso derrubaria a API inteira no boot. Sem credenciais reais, `/auth/google` simplesmente falha no redirecionamento do lado do Google, sem quebrar o resto da aplicação (mesmo espírito do `PaymentProvider` mock: não bloquear por falta de chave real).
- **Asaas real vs. mock**: `AsaasPaymentProvider` implementa a chamada real à API (`/customers`, `/payments`) mas lança `ServiceUnavailableException` explícita se `ASAAS_API_KEY` não estiver setada — diferente do Mock, que sempre aprova. `PAYMENT_PROVIDER=mock` é o default em dev (`.env.example`).
- **Preços**: `PERIOD_DISCOUNT` (mensal 0% / trimestral 8% / semestral 15% / anual 25%) e os valores mensais de cada plano (Essencial 149 / Plus 219 / Elite 349, placeholders) ficam duplicados entre `packages/ui` (preview na landing) e `apps/api/src/payments/payments.service.ts` (cálculo real de cobrança) — a API é a fonte da verdade para o valor cobrado; a landing é só preview de marketing.
- **Fluxo de redirecionamento pós-login**: CTA de plano na landing (`/criar-conta?plano=X&periodo=Y`) → após cadastro/login, `postAuthPath()` manda o usuário para `/planos` com o plano pré-selecionado (fonte de preço real vem da API) → `/checkout` → aprovação automática (mock) libera a assinatura e redireciona para `/app`.
- **`apps/worker/prisma/schema.prisma` duplicado** de `apps/api/prisma/schema.prisma`: cada app roda em container Docker separado (node_modules independentes), então não dá pra compartilhar o Prisma Client gerado sem um pacote comum. Duplicado o schema (o worker nunca roda `migrate`, só lê/escreve via Prisma Client) em vez de criar `packages/db` agora — dívida técnica registrada; extrair para um pacote compartilhado se o schema mudar com frequência nas próximas fases.
## Fase 4 — Onboarding (anamnese única)

- **BullMQ `jobId` não aceita `:`**: `RemindersQueueService` originalmente usava `anamnesis:${userId}` como jobId — quebrava com `Custom Id cannot contain :`. Trocado para `anamnesis-${userId}`. A escrita no Postgres já tinha commitado antes do erro (só o enfileiramento do lembrete falhava), então o bug causava 500 mas sem perda de dado — mesmo assim corrigido antes de seguir.
- **Reagendamento de lembrete a cada save**: em vez de um único lembrete fixo, cada PATCH de rascunho remove o job anterior (`queue.remove(jobId)`) e agenda um novo com `jobId` fixo por usuário — evita empilhar notificações duplicadas enquanto o cliente preenche aos poucos. `submit()` cancela o lembrete pendente.
- **Avaliação física inicial puxada da Fase 7**: o último passo do wizard de anamnese (`escopo.md` §7.1) cria um registro em `Assessment` — implementei um `AssessmentsModule` mínimo (POST/GET) agora em vez de esperar a Fase 7, para não modelar a mesma entidade duas vezes; a Fase 7 só precisa adicionar os gráficos por cima do que já existe.
## Fase 5 — Painel do profissional

- **Cada publicação cria uma nova versão do plano/treino** em vez de editar in-place: `POST /admin/clients/:id/meal-plan` sempre cria um `MealPlan` novo (idem `Workout`); a área do cliente (Fase 6) mostra o mais recente `publishedAt`. Evita lógica complexa de diff/merge em listas aninhadas (refeições→itens, treino→exercícios) para o MVP; histórico de versões fica de graça (todas ficam no banco).
- **Seeds de alimentos/exercícios**: valores nutricionais aproximados de tabelas públicas (TACO/USDA) — marcados como placeholder editável no admin, não são fonte nutricional certificada.
## Fase 9 — APK (Capacitor)

- **Sem Android SDK/Gradle/JDK moderno neste ambiente**: só há JDK 1.8 instalado, sem `ANDROID_HOME`/`gradle`. Buildar o APK localmente não é viável aqui. Em vez de pular a fase, criei `.github/workflows/android-apk.yml` — builda de verdade no GitHub Actions (`ubuntu-latest`, JDK 21, Android SDK 36 via `android-actions/setup-android`) a cada push em `apps/web/**`, publicando o APK debug como artifact. Essa é a validação real de "APK builda" (ver `DECISIONS.md`/`PROGRESS.md` para o link do run).
- **App do cliente empacota o SPA inteiro** (`/app/*` **e** `/admin/*`), não só `/app/*` como o `escopo.md` §4.5 sugere no texto — dividir o bundle exigiria um segundo entry-point/build do Vite. Como `/admin/*` continua protegido por `ProtectedRoute role="PROFESSIONAL"` no cliente **e** por `ProfessionalGuard` na API, um cliente autenticado como `CLIENT` não consegue acessar nada de admin mesmo com o código presente no APK — mitigação aceitável para o MVP; separar os bundles fica como melhoria futura se o tamanho do APK virar problema.
- **Ícone/splash placeholder**: a logo do repo (`logo.webp`) é um wordmark horizontal (2048×439), não um ícone quadrado — inadequado para ícone de app. Gerei um ícone/splash placeholder de marca (fundo `--ink-900`, anel de continuidade em gradiente amarelo, via `apps/web/scripts/generate-app-icons.mjs` + `@capacitor/assets`) até haver um ícone quadrado oficial da marca.
- **Push (FCM)**: estrutura pronta e ativa no app (`@capacitor/push-notifications`, `apps/web/src/lib/push.ts`, roda só em `Capacitor.isNativePlatform()`) e endpoint `POST /client/push-token` salva o token do dispositivo (`User.pushToken`, nova migration). Sem `google-services.json` real (precisa de um projeto Firebase), o registro não dispara — não quebra o app, conforme default do `escopo.md` §13.0 ("ativação em fase posterior").
- **`webDir: "dist"` sem `server.url`**: o app roda com os assets buildados embutidos (offline-first), não carregando a landing/app via URL remota — só chama a API real (`VITE_API_URL` de produção, injetada no build do CI).

- **Validação e2e da Fase 3**: feita via `curl` direto na API rodando localmente contra o Postgres do Docker (cadastro → login → cupom → checkout → assinatura `ACTIVE` confirmada no Postgres). Front (`/criar-conta`, `/planos`, `/checkout`) validado por `vite build` sem erros de tipo; teste de navegador real (Playwright/`claude-in-chrome`) segue pendente pelo mesmo motivo das fases anteriores.

## 2026-07-22 — Mercado Pago configurável pelo admin + Google Pay

- **Apple Pay não incluído**: o Mercado Pago não suporta Apple Pay como método de pagamento no Brasil hoje — confirmado com o usuário. Escopo real ficou só **Google Pay**, via Payment Brick do MP (mostra o botão automaticamente quando o dispositivo/navegador é elegível). Não foi criado nenhum botão "de mentira" nem integrado um segundo gateway (ex. Stripe) só para Apple Pay — custo/complexidade (dois gateways, duas reconciliações) não justificado pelo pedido original.
- **Credencial nunca em `.env`**: diferente do Asaas (`ASAAS_API_KEY` em env var), o cliente não quis compartilhar o Access Token do Mercado Pago com a equipe de dev. Criada a tabela `PaymentSettings` (novo model Prisma) onde o profissional cola o token direto em Admin → Pagamentos; o token é cifrado (AES-256-GCM, `apps/api/src/crypto/crypto.service.ts`) com uma chave mestra (`ENCRYPTION_KEY`, só no `.env` do servidor) e nunca reexibido em texto puro (a tela só mostra "configurado, terminando em ****1234"). `PaymentsService.resolveProvider()` agora é assíncrono e busca o gateway ativo no banco a cada checkout, com fallback para o comportamento legado (`PAYMENT_PROVIDER` env var) se nada estiver configurado — não quebra ambientes já rodando com Asaas/mock.
- **Migration escrita manualmente** (`apps/api/prisma/migrations/20260722120000_add_payment_settings`): não havia Postgres deste projeto rodando neste ambiente para `prisma migrate dev` gerar o SQL automaticamente; escrita à mão seguindo o padrão das migrations anteriores e validada com `prisma validate`/`prisma generate` (schema válido, client gerado, `tsc`/`nest build` sem erro). Ainda precisa rodar de verdade contra um Postgres (`prisma migrate deploy`, já automático no boot do container via `docker-entrypoint.sh`) antes de considerar testado ponta a ponta.
- **API de Payments (`/v1/payments`) em vez da Orders API**: o Mercado Pago vem migrando a documentação para uma "Orders API" mais nova; optei por manter `/v1/payments` (estável, bem documentada, é o que a maioria das integrações usa hoje) em vez de adotar a API mais nova sem conseguir validar contra credenciais reais neste ambiente. Reavaliar se `/v1/payments` for descontinuada.
- **Validação Playwright pendente**: com o MCP do Playwright agora conectado, a tela `/admin/pagamentos` e o fluxo de `/checkout` ainda não foram abertos num navegador real nesta sessão — falta uma `PUBLIC_KEY`/`ACCESS_TOKEN` de teste do Mercado Pago (o cliente ainda não forneceu) para validar o Payment Brick/botão Google Pay de ponta a ponta.
